# FNS-INTEGRATION — интеграция с ФНС

## Текущее состояние: ManualImitationGateway

Все операции с ФНС инкапсулированы за абстракцией `FnsGateway`:

```
web/src/lib/fns/gateway.ts
```

Интерфейс:

```ts
interface FnsGateway {
  readonly id: 'manual' | 'lk-fns' | 'operator'
  readonly label: string
  readonly available: boolean

  getEnsBalance(): Promise<EnsBalance | null>
  requestReconciliation(kind: ReconciliationKind): Promise<SubmitResult>
  submit(docTitle: string): Promise<SubmitResult>
}
```

Единственная активная реализация — `ManualImitationGateway`:

- Пользователь **вводит сальдо ЕНС вручную** (из личного кабинета nalog.ru или через бухгалтера)
- `getEnsBalance()` возвращает введённое значение из `taxOfficeStore`
- `requestReconciliation()` и `submit()` возвращают имитированный `SubmitResult { ok: true, imitation: true }`
- Раздел «Налоговая» (`/tax-office`) показывает статус «Имитация» и предлагает скачать XML для ручной загрузки в ЛК ФНС

---

## Почему нельзя получить данные ЕНС «по ИНН» без авторизации

Сальдо ЕНС, акты сверки, требования об уплате — **налоговая тайна** (ст. 102 НК РФ). Получить их для конкретного ИНН без криптографической авторизации владельца невозможно. Это ограничение не обходится: любой сервис, предлагающий «проверку ЕНС по ИНН» без КЭП, либо показывает публичные данные (не ЕНС), либо действует с нарушением.

## Будущие реализации

### LkFnsGateway — через Личный кабинет ФНС

Авторизация пользователя в ЛК ФНС по его КЭП, серверный слой для криптографических операций.

**Что нужно:**
- Серверный компонент (Node.js / Python на Railway/VPS)
- КриптоПро CSP или ViPNet (ГОСТ 34.10-2012, с 01.04.2026 — ГОСТ 34.12-2018/34.13-2018)
- API ЛК ФНС: авторизация по сертификату → получение токена сессии → ИОН-запросы
- Хранение токена сессии на сервере (не в localStorage браузера — безопасность)

**Что даст:** авто-получение сальдо ЕНС, история платежей, входящие требования.

### OperatorApiGateway — через аккредитованного оператора ЭДО

Контур (Диадок), Астрал, СБИС и другие операторы имеют аккредитацию ФНС и API для:
- ИОН-запросов (информационное обслуживание налогоплательщика): сальдо ЕНС, акты сверки
- Приём и отправка требований
- Отправка отчётности (декларации, уведомления ЕНС)

**Что нужно:**
- Договор с оператором + API-ключ
- Серверный слой для подписи XML документов КЭП клиента
- Маппинг форматов оператора на внутренние структуры (`EnsBalance`, `SubmitResult`)

**Преимущество перед LkFnsGateway:** операторы берут на себя поддержку форматов, не нужно следить за изменениями API ФНС.

## Раздел «Налоговая» (/tax-office) — сейчас

Страница `TaxOffice` работает с `taxOfficeStore` и `ManualImitationGateway`:

- **Сальдо ЕНС** — ручной ввод (дата + сумма). Положительное = переплата, отрицательное = долг.
- **Сверка** — кнопки «Запросить справку о сальдо / о принадлежности / акт сверки». Имитация: создаётся запись в сторе со статусом «Отправлен» → через 3 секунды → «Получен» (демо-ответ).
- **Письма** — входящие требования и ответы. Ручной ввод.

Экран «Отправка отчётности» (в разделах Taxes / UsefulDocs) — при нажатии «Отправить» вызывается `gateway.submit(docTitle)`, который возвращает имитацию с UI-статусом «Отправлено (имитация)» и предлагает скачать XML.

## Как переключить реализацию

При появлении реальных gateway-реализаций переключение происходит в одном месте — провайдер контекста `FnsProvider`:

```tsx
// web/src/providers/FnsProvider.tsx
const gateway = new ManualImitationGateway()   // ← заменить на LkFnsGateway или OperatorApiGateway
```

Все экраны (`TaxOffice`, `Reports`, `Taxes`, `UsefulDocs`) получают gateway через контекст и не знают о реализации.
