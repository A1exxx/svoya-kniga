# Серверный режим: аккаунты + настоящая база данных

Этот документ — как перевести «СвояКнигу» из локального режима (данные в браузере)
в **боевой**: вход по логину/паролю, данные в надёжной базе на сервере, многослойные
бэкапы, доступ откуда угодно, переносимость.

Фундамент уже собран и протестирован (`backend/app/{db,models,security,auth,sync}.py`,
7 тестов в `tests/test_auth_sync.py`). Не хватает только **выбора хостинга** — это
требует вашего аккаунта, денег и решения по закону (см. ниже).

---

## ⚠️ Главное про закон (152-ФЗ) — прочитать до первых реальных данных

ИНН, СНИЛС, паспорта и зарплаты сотрудников клиентов — это **персональные данные
граждан РФ**. По ст. 18 ч. 5 152-ФЗ их **первичная база обязана физически
находиться на территории РФ**.

- Для **боевого режима с реальными данными** — только **РФ-хостинг** (Timeweb,
  Yandex Cloud, Selectel, VK Cloud, Cloud.ru). Зарубежные (Supabase, Neon, Render,
  Railway, AWS) — **нельзя** для реальных ПДн.
- Зарубежный/бесплатный хостинг можно использовать **только для теста с
  выдуманными данными** (никаких настоящих ИНН/ФИО).
- Оператор ПДн (ИП/сервис) обязан **подать уведомление в Роскомнадзор**
  (pd.rkn.gov.ru) **до начала** обработки. Штраф за неуведомление с 30.05.2025 —
  до 300 000 ₽; за хранение базы за рубежом — до 6 млн ₽.

**Вывод:** тестировать механику — можно бесплатно за рубежом на тестовых данных;
как только заводим реальных клиентов — РФ-хостинг + уведомление РКН.

## Архитектура

```
Браузер (фронтенд на GitHub Pages, без изменений в локальном режиме)
        │  HTTPS, cookie-сессия (httpOnly)
        ▼
FastAPI (backend/app) ── аккаунты, версии рабочего стола, экспорт
        │
        ▼
PostgreSQL (РФ managed) ── данные + история ревизий + PITR
```

- **Аккаунты:** email + пароль (Argon2id, OWASP-профиль). Сессия — opaque-токен
  в httpOnly-cookie + серверная таблица `sessions` (можно отозвать = выйти на всех
  устройствах). Защита от перебора — лимит 5 попыток / 15 мин.
- **Хранилище:** один рабочий стол на пользователя = весь кабинет (ИП, операции,
  документы, сотрудники) как JSON. **Каждое сохранение — новая неизменяемая
  ревизия** (`workspace_versions`) → можно посмотреть историю и откатиться.
- **API:** `/api/auth/{register,login,logout,me}`, `/api/workspace` (GET/PUT),
  `/api/workspace/versions`, `/api/workspace/restore/{v}`, `/api/workspace/export`.

## «Чтобы ничего не потерялось» — 5 слоёв бэкапа

1. **PITR** у managed-провайдера — откат к любой секунде (WAL-архив).
2. **Автоснапшоты БД** провайдера — ежедневно, хранятся 7–14 дней.
3. **`backup.sh`** — ежедневный `pg_dump` в файл/объектное S3-хранилище (внешняя копия).
4. **История версий** в самой БД — на каждое сохранение рабочего стола.
5. **Кнопка «Скачать архив»** в интерфейсе — пользователь забирает свои данные.

## Переносимость

Стандартный PostgreSQL 15/16 + Alembic-миграции. Переезд между провайдерами:
`pg_dump -Fc` → `pg_restore` + смена `DATABASE_URL`. Никакого vendor-lock.

## Запуск

### Локально (SQLite, ноль настроек)
```bash
cd backend
.venv/Scripts/python -m uvicorn app.main:app --reload --port 8000
# http://localhost:8000/docs
```

### Локально как «боевой» (Postgres в Docker)
```bash
cd backend
docker compose up --build      # API на :8000, Postgres на :5432
```

### Прод (РФ, ориентир Timeweb Cloud — самый дешёвый старт)
1. Создать **Managed PostgreSQL** (от ~790 ₽/мес).
2. Развернуть контейнер из `backend/Dockerfile` (App Platform / VPS).
3. Переменные окружения: `DATABASE_URL` (Postgres), `COOKIE_SECURE=true`,
   `COOKIE_SAMESITE=none` (если фронт на другом домене), `CORS_ORIGINS` (адрес фронта).
4. HTTPS (Let's Encrypt / прокси провайдера).
5. Cron: `backup.sh` раз в сутки.
6. Подать уведомление в РКН.

**Ориентир по деньгам:** боевой РФ-вариант ≈ **1 200–1 500 ₽/мес**
(Postgres + контейнер + хранилище бэкапов). Тест за рубежом на фейковых данных — бесплатно.

## Подключение фронтенда (следующий шаг)

Фронтенд получит экран входа и синхронизацию за флагом `VITE_API_BASE`:
- флаг не задан → текущий локальный режим (демо на GitHub Pages работает как есть);
- задан адрес API → появляется вход, кабинет грузится/сохраняется на сервере,
  локальный IndexedDB остаётся офлайн-кэшем.
