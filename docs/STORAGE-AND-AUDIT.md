# STORAGE-AND-AUDIT — хранение данных и журнал

## Модель хранения: localStorage

Все данные хранятся в `localStorage` браузера. Ключи сгруппированы по префиксу `svoyakniga.` и скоупированы по `orgId` (ИНН или внутренний идентификатор ИП):

| Ключ localStorage | Стор | Что хранит |
|-------------------|------|-----------|
| `svoyakniga.org.<orgId>` | `orgStore` | Реквизиты ИП (ИНН, ОГРНИП, адрес, ОКТМО, НДС-ставка) |
| `svoyakniga.ops.<orgId>` | `opsStore` | Операции доходов и расходов |
| `svoyakniga.docs.<orgId>` | `docsStore` | Документы (счета, акты, накладные, УПД) |
| `svoyakniga.contractors.<orgId>` | `contractorsStore` | Контрагенты |
| `svoyakniga.goods.<orgId>` | `goodsStore` | Номенклатура |
| `svoyakniga.employees.<orgId>` | `employeesStore` | Сотрудники |
| `svoyakniga.archive.<orgId>` | `archiveStore` | Архив задач и событий |
| `svoyakniga.taxoffice.<orgId>` | `taxOfficeStore` | Данные раздела «Налоговая» |
| `svoyakniga.paramOverrides.v1` | `paramsStore` | Пользовательские переопределения YearParams (глобальный ключ, не скоупирован по orgId) |

Служебные ключи (не содержат бизнес-данных):

| Ключ | Назначение |
|------|-----------|
| `svoyakniga.snapshots.v1` | Кольцевой буфер снимков (JSON, до 20 штук) |
| `svoyakniga.audit.v1` | Журнал действий (до 500 записей) |
| `svoyakniga.dadata.token` | Токен DaData для автозаполнения по ИНН |
| `svoyakniga.lastAutoSnapshot` | Дата последнего авто-снимка (ISO) |

## Снимки / откат / бэкап / журнал

Модуль: `web/src/lib/storage/storeAdmin.ts`

### Снимки (Snapshots)

```ts
createSnapshot(label: string): Snapshot   // сохраняет все data-ключи
restoreSnapshot(id: string): void         // восстанавливает все ключи из снимка
listSnapshots(): Snapshot[]
deleteSnapshot(id: string): void
```

- Снимок захватывает **все ключи** с префиксом `svoyakniga.*`, кроме служебных (`snapshots`, `audit`, `dadata.token`, `lastAutoSnapshot`).
- Кольцевой буфер: хранится не более **20 снимков**. При переполнении удаляется самый старый.
- **Авто-снимок:** при первом открытии страницы после 00:00 (если прошло ≥ 24 часа с последнего авто-снимка) автоматически создаётся снимок с меткой `auto-YYYY-MM-DD`.

### Бэкап / импорт

```ts
exportBackup(): string   // JSON-строка всех data-ключей → скачивается как .json
importBackup(json: string): void   // восстанавливает все ключи из JSON
```

Экспорт включает все бизнес-данные, кроме токенов и метаданных снимков.

### Журнал действий

```ts
logAudit(action: string, detail: string): void
listAudit(): AuditEntry[]
```

Формат записи: `{ id, at: ISO, action, detail }`. Кольцевой буфер **500 записей**. Действия логируются автоматически при создании документов, редактировании реквизитов, изменении параметров налогов, операциях с сотрудниками.

### Экран Admin (`/admin`)

Позволяет:
- Просматривать и восстанавливать снимки
- Скачать / загрузить полный бэкап
- Просматривать журнал действий (фильтр по типу и дате)
- Оценить занятое место в localStorage

## Будущее: переход на IndexedDB + сервер

Текущий localStorage-подход ограничен:

- ~5–10 МБ на домен (зависит от браузера)
- Данные привязаны к конкретному браузеру и устройству
- Нет per-field истории изменений

**Следующий этап:**

1. **IndexedDB** — увеличивает лимит хранилища до нескольких ГБ, поддерживает транзакции и индексы. Позволяет реализовать per-field журнал (каждое изменение поля → отдельная запись с timestamp и пользователем).
2. **Серверная база** (Railway / VPS, PostgreSQL) — многопользовательский доступ, история изменений, облачный бэкап.
3. **Синхронизация** — offline-first: IndexedDB как локальный кэш, сервер как источник правды. Конфликты разрешаются по timestamp + last-write-wins (или явный merge-экран).

Этот переход связан с появлением многопользовательского доступа и реальной интеграции с ФНС (см. `docs/FNS-INTEGRATION.md`).

Интерфейс сторов (`getOrg`, `saveOrg`, `listOps`, ...) при переходе не меняется — только реализация под капотом переключается с `localStorage` на `IndexedDB` / `fetch`.
