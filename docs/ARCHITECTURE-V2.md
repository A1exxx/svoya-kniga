# СвояКнига V2 — архитектура, дебаты и дорожная карта «круче Эльбы»

> Дата: 2026-06-20. Основа: два глубоких ресёрча (конкуренты РФ-бухгалтерий; форматы ФНС и
> open-source) + реализация мультипользовательского режима. Откат к прошлой версии:
> `git checkout backup-2026-06-20-pre-v2-multiuser`.

## 1. Главный рыночный вывод (из конкурентного анализа)

**Ни один продукт на рынке РФ не совмещает в одном приложении:**
- простоту и цену розничного сервиса для ИП (Контур.Эльба, Т-Банк, Точка), и
- полноценный мультиклиентный движок для обслуживающих бухгалтерий
  (Контур.Экстерн «Обслуживающая бухгалтерия», 1С:Мультибух — отдельные дорогие B2B-продукты).

У Эльбы до 100 организаций в аккаунте, но задачи всех клиентов свалены в один список,
без сводной панели, без явных ролей на клиента. **СвояКнига закрывает ровно этот пробел:**
простота Эльбы + движок Экстерна в одном продукте. Анти-цель: UX СБИС («непонятный ад»
по отзывам) — мощность не должна убивать простоту.

## 2. Дебаты по ключевым решениям (что выбрали и почему)

### 2.1 Модель данных на сервере: JSON-снимок vs полная нормализация
- **За нормализацию** (операции/документы/сотрудники в таблицах): SQL-отчёты по всем клиентам,
  частичные обновления, меньше трафика.
- **За версионируемый JSON-снимок** (текущее): каждая правка — неизменяемая ревизия (защита
  «ничего не потерялось» — главное требование), формат = локальный бэкап (один код),
  офлайн-first (фронт полностью работает без сервера), тривиальная миграция SQLite→Postgres.
- **Решение: снимок + ревизии остаётся ядром**, поверх — реляционные таблицы аккаунтов/доступов
  (users, workspaces, memberships, invites, sessions, audit_log). Нормализация операций — только
  когда появится реальная потребность (кросс-клиентские SQL-отчёты на сервере); путь миграции:
  парсер снимка → таблицы, без изменения фронта.

### 2.2 Права: роль на весь кабинет vs права на конкретного клиента (ИП)
- Экстерн даёт гранулярные права «по организациям». У нас кабинет (workspace) = портфель ИП.
- **Решение (сейчас):** роль на кабинет — `owner` (всё + команда), `accountant` (полный доступ
  к данным), `viewer` (только чтение, сервер отклоняет запись 403). Просто и понятно.
- **Roadmap:** назначение бухгалтера на конкретных ИП внутри кабинета (ACL на org-уровне) +
  «загрузка по сотрудникам» — фича, которой нет ни у кого из конкурентов в рознице.

### 2.3 Приглашения: e-mail-инвайты vs код
- **Решение: одноразовый 8-символьный код** (алфавит без 0/O/1/I/L — диктуется по телефону),
  TTL 7 дней. Не требует почтового сервера (локальный этап!), работает и на проде.

### 2.4 Кто «владелец данных» при работе бухфирмы
- Вариант А: кабинет принадлежит бухфирме, клиенты — записи внутри.
- Вариант Б: кабинет принадлежит клиенту-ИП, он приглашает бухгалтера.
- **Решение: поддерживаем ОБА** одной механикой memberships: бухфирма может вести портфель
  в своём кабинете (все ИП внутри), а «продвинутый» клиент — завести свой кабинет и позвать
  бухгалтера кодом. Модель не навязывает организацию работы.

### 2.5 Локальный этап без потери пути на сервер
- Всё работает на одном ПК: FastAPI + SQLite (`backend/svoyakniga.db`) отдаёт и приложение
  (dist-local), и API с одного адреса (без CORS). Те же таблицы и код — на Postgres в проде
  (переменная DATABASE_URL). Ничего переписывать при переезде не нужно.

## 5. Дорожная карта «круче Эльбы» (приоритизировано ресёрчем)

**P1 — бухфирма (частично сделано):**
1. ✅ Единый дашборд по всем клиентам («Обзор клиентов»).
2. ✅ Роли owner/accountant/viewer + свой логин у каждого.
3. ✅ Автор каждой правки в истории версий.
4. ⬜ Назначение бухгалтера на конкретных ИП + загрузка по сотрудникам (нет ни у кого).
5. ⬜ Массовые операции по всем клиентам разом (сформировать все уведомления квартала одной кнопкой).
6. ⬜ Авто-аудит качества учёта по каждому клиенту (отчёт об ошибках: пустые реквизиты, разрывы
   нумерации счетов, операции без документов) — вдохновение 1С:Мультибух.

**P2 — оба сегмента:**
7. ⬜ Остаток денег на счету (не только доходы/расходы) — главная жалоба на Эльбу.
8. ⬜ Калькулятор выгодной ставки НДС 5/7/22% под обороты клиента.
9. ⬜ Автоучёт комиссий маркетплейсов (WB/Ozon/ЯМ) в УСН.
10. ⬜ Уведомление о приближении лимита 20 млн (порог НДС 2026) и 450 млн (слёт с УСН).
11. ⬜ Простой/полный режим интерфейса (переключатель).
12. ⬜ Проверка контрагентов (ЕГРЮЛ/арбитраж) в карточке — расширение DaData-интеграции.

**Серверный этап (после утверждения функционала):** VPS РФ (~650 ₽/мес, Selectel) + Postgres +
домен; отправка в ФНС через оператора (Такском «НП++» ~6 620 ₽ старт — выбор пользователя, или
Астрал.Платформа API «оплата за результат»); облачная КЭП; МЧД для представителя. Всё описано в
`docs/REMOTE-CLIENTS-EDO.md` и `docs/SERVER-SETUP.md`.

## 6. Границы честности (не имитировать понимание)
- Реальная отправка в ФНС и автоподтяжка сальдо/требований — только через оператора ЭДО с КЭП
  (налоговая тайна): в приложении это имитация со статусами до серверного этапа.
- Машиночитаемый 2D-штрих-код бланков печатает «Налогоплательщик ЮЛ»/ЛК из нашего XML.
- ОСНО — базовый модуль (НДФЛ+НДС+взносы); 3-НДФЛ и КУДиР по 86н — в развитии.
